Human Context
Behavior

고객은 요금제·혜택·단말 상태를 각각 확인하며 스스로 최적안을 찾기 위해 여러 화면을 반복 탐색합니다.

Problem (AS-IS)

데이터가 존재함에도 종합 해석이 없어 개인화 제안이 부재합니다. 고객이 직접 계산해야 하는 부담이 발생합니다.

Target (TO-BE)

사용 패턴·상황·데이터를 입체 해석해 단일 최적안을 선제 제시합니다. 절감액 등 실질 가치 근거와 함께 즉시 판단 가능한 상태를 만듭니다.

Rules
DCD_1_R1
고객 데이터를 기반으로 단일 최적안을 선별해 제안해야 합니다. 복수 옵션 나열 금지.
IF / THEN
  • IF usage_pattern AND contract_status resolved THEN render single OptimalPlanCard; suppress option list
  • IF plan_change_penalty == 0 THEN append NoPenaltyLabel to OptimalPlanCard
  • IF saving_amount > 0 THEN render SavingLabel showing concrete monthly/annual value
  • IF device_change == true AND bundle_complex == true THEN render single pre-configured BundleRecommendationCard (not a list)
✓ DO
  • 고객 사용 패턴·상황·데이터를 반영해 결과를 우선 제시하고 고객 판단을 유도합니다.
  • 조건이 복잡한 요금제 변경 시, 데이터 기반 단일 최적 옵션을 먼저 제시합니다.
  • 단말기 변경과 같이 결합이 복잡한 경우, 한눈에 이해 가능한 최적 구성을 제공합니다.
✗ DON'T

IF / THEN

  • IF benefit_value_type == "abstract" THEN convert to concrete_value_expression (e.g. "커피 1잔", "영화 티켓 2매")
  • IF plan_change_no_penalty == true THEN render NoPenaltyBadge with copy: "바로 변경해도 위약금이 발생하지 않아요."
✓ DO
  • 절감 금액, 혜택 변화 등 정량 근거를 포함합니다.
  • 가치 판단이 필요한 경우 실질 금액 또는 공감 표현으로 변환합니다.
  • 고객이 충분히 이해하고 선택을 증명하는 맞춤 안내를 명확히 제공합니다.

Human Context
Behavior

고객은 요금제와 상품이 나열된 상태에서 각 항목을 직접 계산하고 기억하며 판단합니다. 반복 탐색과 재확인이 발생합니다.

Problem (AS-IS)

동일한 나열 구조로 인해 고객이 스스로 비교 기준을 만들어야 합니다. 현재 상태 대비 변화를 파악하기 어려워 결정이 지연됩니다.

Target (TO-BE)

현재 계약 기준으로 변화되는 핵심 요소만 대조합니다. 고객은 '무엇이 어떻게 달라지는지'만 확인하고 즉시 결정합니다.

Rules
DCD_2_R1
비교 결과는 개별 항목 나열이 아닌 직관적 대조(delta) 형태로 제공해야 합니다.
IF / THEN
  • IF comparison required THEN render delta_view showing change_amount + change_direction (not raw absolute values)
  • IF data_allowance increases THEN render DeltaLabel: "데이터 +{data_delta}GB"
  • IF monthly_cost decreases THEN render DeltaLabel: "월 -{saving_amount}원 절약"
  • IF complex_conditions == true THEN render key_change_summary alongside full ComparisonView
✓ DO
  • 현재 대비 변화값(delta)을 중심으로 즉시 이해 가능하게 표현합니다.
  • 복잡한 조건 비교 시 변경되는 핵심 정보만 요약한 구조를 병행 제공합니다.
✗ DON'T
triggerrejectionfallback

DCD_2_R2
의사결정에 필요한 핵심 정보만 선별해 제공합니다. 불필요한 세부 정보는 숨깁니다.
IF / THEN
  • IF full_spec_item_count > critical_info_threshold THEN surface only: cost_delta, data_delta, benefit_delta; hide: full_terms, footnotes
  • IF user requests detail THEN progressively reveal full_spec on user action (not by default)
✓ DO
  • 의사결정에 필요한 핵심 정보만 선별해 제공합니다.
✗ DON'T
triggerrejectionfallback
비교 화면 초기 렌더링 불필요 세부 조건 전체 노출 핵심 3종(비용/데이터/혜택 변화)만 선별 표시
DCD_2_R3
user_segment에 따라 비교 방식을 다르게 구성해야 합니다. 동일 구조 일괄 적용 금지.
IF / THEN
  • IF user_segment == "value_type" THEN lead ComparisonView with benefit_upgrade_description; show concrete value gain first
  • IF user_segment == "analysis_type" THEN lead ComparisonView with data_driven_rationale; show saving_reason before saving_amount
  • IF user_segment == "relationship_type" THEN lead ComparisonView with group_benefit_changes; highlight family impact
  • IF user_segment == "senior" THEN use simplified_language_template; show single data_delta label only; suppress benefit_list
  • IF user_segment == null ELSE default to "value_type" presentation
✓ DO
  • user_segment 별로 비교 방식과 리드 카피를 다르게 구성합니다.
✗ DON'T
triggerrejectionfallback
