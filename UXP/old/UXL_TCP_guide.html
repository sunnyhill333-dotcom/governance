UXL_TCP_1

6-1. 종합적인 처리 결과 안내

여러 단계를 거쳐 진행된 개별 태스크들을 하나의 완성된 흐름으로 연결해 안내함으로써, 고객이 처리 내역을 직관적으로 인지하도록 해야 합니다.

BEHAVIOR

고객은 여러 단계를 거쳐 과업을 완료한 뒤에도 각각의 처리 결과를 개별적으로 확인하며 전체 상태를 재구성하려 합니다. 결과가 분절되어 제공되기 때문에 전체 변화 맥락을 스스로 조합해야 하며, 완료 이후에도 이해를 위한 추가 탐색이 발생합니다.

AS-IS

단계별 완료 메시지가 나열되어 있어 전체 변화 맥락을 조합해야 하며, 완료 이후에도 이해를 위한 추가 탐색이 발생합니다.

TO-BE

시스템이 모든 처리 결과를 하나의 결과 흐름으로 재구성하여 제공함으로써, 고객이 완료 상태를 직관적으로 인지하도록 만들어야 합니다. 이는 단계 단위가 아닌 '결과 단위'로 정보를 통합해야 한다는 의미입니다.

pattern_id UXL_TCP_1

trigger task.completion_status == 'completed'
output_component TaskCompletionSummary → slot: completion_primary, state: result_unified
input_data completed_task_type (session_state), task_result_summary (real_time_api)
suppressed_when task_result_summary unavailable → fallback: generic_completion_message with task_type label

UXL_TCP_2

6-2. 핵심 정보 구조화

복잡한 설명이나 텍스트 나열을 최소화하고, 고객이 알아야 할 혜택 변화와 주요 수치를 중심으로 시각화하여 고민 없이 변화된 가치를 파악하도록 제공해야 합니다.

BEHAVIOR

고객은 완료 후 제공된 정보가 나열된 형태로 제공되면서 핵심 지표를 선별하고 해석하는 부담이 발생하고, 이는 의사결정의 피로로 이어집니다.

AS-IS

수치 나열과 설명 중심 문장으로 인해 고객이 직접 해석 과정을 거쳐야 합니다. 이로 인해 핵심 가치 전달 실패 및 추가 문의가 발생합니다.

TO-BE

고객의 판단이 아닌 시스템의 해석 결과를 시각화하는 구조로, 고객이 실제로 체감할 수 있는 가치(비용 변화, 혜택 증감 등)와 핵심 정보만 구조적으로 시각화해야 합니다.

input_data usage_data, benefit_delta, comparison_data (all real_time_api, optional)
suppressed_when visualization_data unavailable → render text-based completion summary without chart

UXL_TCP_3

6-3. 사후 가이드 제공

완료 후에도 필요한 후속 조치나 유의사항을 선제적으로 안내하여, 불필요한 재탐색 없이 경험을 마무리해야 합니다.

BEHAVIOR

고객은 과업 완료 이후 필요한 후속 행동이나 주의사항을 확인하기 위해 다시 탐색하거나 문의합니다.

AS-IS

완료 이후의 상태 변화에 대한 안내가 부족하여 고객이 다음 행동을 스스로 판단해야 하며, 이로 인해 불필요한 재탐색이 발생합니다.

TO-BE

고객의 의도를 다시 읽고 다음 태스크를 이어서 시스템이 선제적으로 안내해야 합니다. 고객이 특정 태스크를 수행한 경우, 이후 진행하면 좋은 다음 단계를 고객이 인지하지 못하는 상황이 없도록 해야 합니다.

suppressed_when followup_actions unavailable → render completion page without followup section
