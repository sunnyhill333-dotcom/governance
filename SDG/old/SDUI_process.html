02 — 항목별 역할 정의
정책
정책서
Policy Documents — usecase_US-*.md / scenario_*.json
Hard Lock
역할
무엇을 만들 것인가를 결정하는 유일한 기준. 화면 수, 플로우, 기능 입출력, 예외 처리, 정책 규칙을 확정한다. 이후 모든 거버넌스는 이 범위 안에서만 동작한다.
SB 생성 시 제공하는 것
유즈케이스 → 프로세스 → 기능(처리 흐름) : 어떤 화면이 몇 개 필요한지, 어떤 순서로 이어지는지, 각 화면에서 무슨 기능이 실행되는지 정의한다.
목업 생성 시 제공하는 것
기능(실패·예외 케이스) + Policy Item : 어떤 오류 상황이 존재하는지, 각 텍스트·레이블의 정책 기준이 무엇인지 정의한다.
거버넌스와의 관계
거버넌스가 정책서와 충돌할 경우 정책서가 항상 우선한다. 거버넌스는 정책서가 정한 범위 안에서 경험 품질을 높이는 역할만 한다.
UXP
UXP (UX원칙)
UXL_ETR / EXP / SCH / DCD / ACT / TCP / RSV
SB 생성 인풋
역할
이 화면이 속한 여정에서 사용자에게 어떤 경험을 제공해야 하는가를 정의한다. 화면 수나 플로우는 정책서가 결정하지만, 각 화면 안의 정보 위계·섹션 우선순위·상호작용 원칙은 UX원칙이 결정한다.
SB 생성 시 작용 방식
화면 ID가 속한 여정(진입·탐색·결정·실행·완료·CS)을 파악 → 해당 UXL 파일의 DO/DON'T 원칙을 적용해 오가니즘 배치 순서, 조건부 섹션 노출 기준, 정보 밀도를 결정한다.
활용 예시
ACT 여정 화면 → UXL_ACT "입력·선택 간소화" 원칙 적용 → 필수 입력 섹션만 기본 노출, 조건부 섹션은 서버 제어로 처리
UXPT
UXPT (UX패턴)
UX_Pattern_1(구조) / UX_Pattern_2(상태)
목업 생성 인풋
역할
각 UI 컴포넌트가 어떻게 동작해야 하는가를 정의한다. 내비게이션, 버튼, 화면 전환, 로딩, 에러, 리커버리 등 공통 상태 처리 방식을 규정한다.
목업 생성 시 작용 방식
정책서의 기능(실패·예외 케이스)를 읽어 → 어떤 상태 처리 패턴을 써야 하는지 결정한다.

필수 미동의 → 인라인 에러 패턴
버전 불일치 → Recovery Flow 패턴
API 저장 실패 → 시스템 에러 패턴
활용 예시
SB에 정의된 오가니즘이 로딩 상태를 가질 경우 → UX패턴 2-1(스켈레톤 UI) 기준으로 렌더링 방식 결정
VOT
VOT
Voice & Tone — 보이스톤 가이드
목업 생성 인풋
역할
화면 안의 모든 텍스트가 어떤 톤으로 작성되어야 하는가를 정의한다. 버튼 라벨, 안내 문구, 에러 메시지, 추천 카피 등에 적용된다.
목업 생성 시 작용 방식
정책서의 Policy Item(텍스트·안내 규칙)을 읽어 → 해당 상황(안내·추천·경고·관계)에 맞는 VOT 기준을 적용해 카피를 완성한다.
활용 예시
약관 동의 화면의 필수 약관 레이블 → VOT 안내 기준 적용 → "서비스 이용약관 (필수)" 표기 방식·어조 결정
SDG
SDG
SDUI Governance — 컴포넌트 구조 기준
목업 생성 인풋
역할
SDUI에서 컴포넌트를 어떻게 명명하고 구조화하고 페이지를 어떻게 조합하는가의 기준을 정의한다. 오가니즘 ID 체계, 서버 제어 항목 구조, 페이지 단위 조합 규칙을 포함한다.
목업 생성 시 작용 방식
SB에 정의된 오가니즘 ID(ogn-MBR-terms-agree 등)가 SDG 네이밍 기준을 따르는지 검증하고, 서버 제어 항목 구조가 SDG 페이지 조합 규칙에 맞게 설계되었는지 확인한다.
활용 예시
조건부 섹션(최소=0, 최대=1) → SDG 서버제어 항목 "노출 여부" 필드 구조 기준으로 payload 스펙 정의

변하지 않는 단 하나의 원칙
정책서는 기본값이다. 충돌이 발생했을 때 판단이 서지 않으면 정책서를 따른다. 단, 정책서를 따른 결과가 사용자에게 실질적 혼란·이탈·오류를 유발한다고 판단될 경우, 거버넌스는 정책서 수정을 제안할 권한을 가진다. 어느 방향으로 결론이 나든, 충돌을 묵인하거나 기록 없이 넘어가는 것은 항상 금지다. 모든 충돌은 반드시 검수 리포트에 남기고 의사결정 이력을 보관한다.
